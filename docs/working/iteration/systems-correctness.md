> **状态（2026-06-21）**：**R1–R21 全部 ✅ FIXED + 已提交**。R21 = [`LOG.md`](LOG.md) 的 **F174**（head-of-line blocking），由有界 Advance worker 池修复（[ADR 0007](../../decisions/0007-scheduler-async-advance-pool.md)）。

> **第二轮（2026-07-17）**：**R22 + R23 + R24 ✅ FIXED**；**R25 📋 OPEN**（同轮发现、当前休眠、刻意不修）。R24 顺带补上了 R22 自认「不可约减」的窟窿——`-timeout` panic 那轮的孤儿进程与临时目录，**进程内确实没救，但下一轮够得着**。三条已修的**都不是静态猎手扫出的**，都由**用户实机取证**触发——R22 来自「31 个 llama-server 吃掉 12.23G / 16G，机器发烫卡顿」，R23 来自「`make -C backend testend` 连续三次撞 30m 预算」后用户一条 `du -sh` 量出 645MB 缓存。**三条现场假说全被实测证伪**（CPU 争用 / 并发跑法 / R22 引入），**发起 R23 时的两条现场假说亦然**（「今晚才喂进缓存」→ 实为 06-26；「R10–R12 扫时缓存是空的」→ 扫时已 458MB）——见 R23 证据 (5)(6) 与下方教训（一）。**方法论**：归因必须由**实测**定罪，自洽的因果故事在物理证据前一文不值。
> **首轮的盲区已定性，并已被二轮两次出血证实**：11 个猎手只扫 `backend/` 生产码，**`testend/` 座架从不在扫描面内**——首轮唯一提到测试的那句是 R2 的「test runs, `kill -9`, crashes, IDE restart」，把测试跑**当作后端必须扛住的环境危害**（输入），从没反过来问「发出这些硬杀的座架自己有没有收容义务」。R22（进程收容）与 R23（磁盘放大）是**同一个盲区的两次出血**：R10–R12 那轮**专扫 `disk-io-amplification`**，在 `backend/` 里挖出三条（全部真实、全部已修），而**全仓最大的磁盘放大器**——每轮 ~139GB 拷贝——就躺在 `testend/harness/server.go` 第 136 行，没有任何猎手打开过那个文件。**方法论教训（一）**：把某物列为**威胁模型的一部分**，就等于把它排除出了**审查对象**——盲区不在被扫的代码里，在扫描面的定义里。
> **方法论教训（二，R23 新增）——静态扫描对「状态相关成本」结构性失明**：即便 `testend/` 当初在扫描面内，读码也未必挖得出 R23。`cp -R $cache/sandbox $dataDir/sandbox` 这行**源码里不含任何量级信息**：它有多贵，取决于 `~/.anselm-testend-cache` 在**用户磁盘上长到了多大**，而那是运行时状态、不是代码事实。且该状态**单调增长且由本套件自己喂**（`saveRuntimeCache` 懒填充）——于是形成「**缓存越好用、套件越慢**」的反向激励：预置机制越成功地免掉下载，每个用例要拷的字节就越多。这类缺陷只有**真机 + 特定状态**才现形（与 R2 同类：代码读着没问题，只有崩溃/硬杀那一刻才暴露）。**对策**：凡「拷贝/读取一个**由运行时累积**的路径」的代码，静态审查必须问「这东西在真机上会长到多大」并**实测**，而不是读完 `cp -R` 觉得无害就放行。

**Background bash processes (and their whole process trees) are never reaped on shutdown — Manager.Stop() is unreachable**

- **位置**: `backend/internal/bootstrap/build_services.go:213`

## R5 [HIGH] (channel-deadlock) — ✅ FIXED

**Durable SSE Publish on a never-cancelling Detached context blocks forever behind one wedged (connected-but-not-reading) client, freezing the whole workspace stream bus**

- **位置**: `backend/internal/infra/stream/bus.go:106-113 (durable fan-out); trigger paths: backend/internal/app/chat/emit.go:91-107 + backend/internal/app/chat/host.go:141-158; backend/internal/app/notification/notification.go:86-90; backend/internal/pkg/reqctx/workspace.go:61-63`
- **反模式**: unbuffered/bounded send under a lock with no escape when the publisher ctx never cancels (send to a slow/stuck-but-present receiver)
- **触发/累积**: A single SSE client that is connected but not draining: a frozen/backgrounded Flutter desktop window, a suspended laptop, a paused client-side debugger, or just a client on a slow/stalled link. The frontend keeps all three SSE streams open for the entire session and they are 'never idle' (build.go:182 comment), so the wedged-subscriber window is long-lived. Any assistant turn finishing (WriteFinalize → durable message_stop on Detached ctx) or any notification/entity durable frame then blocks under st.mu.
- **证据**: Bus.Publish's DURABLE path fans out under st.mu (the per-workspace lock) with exactly three escapes:

      select {
      case s.ch <- env:
      case <-s.done:        // only closed when the subscriber's HTTP ctx is Done
      case <-ctx.Done():    // the PUBLISHER's ctx
      }
  }
Open/Close/non-ephemeral Signal are durable (frame.go:85-88). The subscriber channel cap is sseBufSize(256)+subscriberHeadroom(256)=512 (subscribe.go:32, build_data.go:89). The SSE pump (response/sse.go StreamSSE) drains `items` and calls flusher.Flush() inline; if the client socket is write-blocked (TCP zero-window) the pump blocks IN Flush() and stops selecting on `items`, so the 512-slot channel fills and never drains. At that point a durable Publish can only proceed via `<-s.done` (client fully disconnects → r.Context() Done → subscribe.go:74-78 cancel → close(done)) or `<-ctx.Done()` (publisher ctx). The lethal combination: MANY durable publishes run on reqctxpkg.Detached() contexts, which are built from context.Background() and DELIBERATELY never cancel (workspace.go:57-62 comment: 'from context.Background() (NOT WithoutCancel) … so the parent request's cancel cannot interrupt it'). Concrete hot path: chatHost.WriteFinalize (host.go:143) runs on a Detached ctx and calls emitMessageStop → publishFrame → Bus.Publish of a Close frame (emit.go:91-107) — a DURABLE send. With a wedged client, this send has no live escape: subscriber ch is full, s.done not closed (still connected), ctx.Done never fires (Detached). It blocks holding st.mu, so EVERY other Publish to that workspace (chat deltas via loop/emit.go, entity build frames, all notification pushes) serializes behind st.mu and also blocks. notification push (notification.go:86) is likewise a durable Signal and is emitted from background/detached paths.
- **修法**: Durable fan-out to a per-subscriber channel must not be able to block the shared producer indefinitely. Options: (1) give the durable send a bounded timeout/escape that is independent of the (possibly-Detached) publisher ctx — e.g. select on s.ch, s.done, AND a time.After(writeDeadline); on timeout, mark that subscriber as lagging and close its done (force-disconnect the slow client) so the bus self-heals, mirroring the SEQ_TOO_OLD recovery the client already handles. (3) Never publish durable frames on a context that cannot cancel: thread a bounded-timeout context (context.WithTimeout off Detached) into WriteFinalize/notification.push so ctx.Done() is a guaranteed escape. Option (1)+(2) together is the robust fix: bound per-subscriber send time and don't hold the workspace lock across a network-paced send.

## R6 [MEDIUM] (subprocess-reaping) — ✅ FIXED

**Boot crash-recovery kills only the recorded parent PID, orphaning uvx/npx grandchildren (python/node)**

- **位置**: `backend/internal/app/sandbox/restore.go:56-73 (killIfAlive, esp. 69)`
- **反模式**: process-group leak on recovery (positive-PID kill instead of negative-PGID kill)
- **触发/累积**: Backend crashes/SIGKILL while a stdio MCP server launched via uvx/npx is resident (common: any installed python/node MCP server stays connected for the whole session). On the NEXT boot, recovery kills the wrapper but not the server grandchild. Each hard-restart cycle that happens while an uvx/npx MCP server is alive can leak one grandchild. Matches the seed's "several MCP servers running" orphans surviving across a long session.
- **证据**: Normal-path kills use the correct pgroup primitive: longLivedHandle.Kill -> killProcessGroup -> `syscall.Kill(-cmd.Process.Pid, SIGKILL)` (infra/sandbox/spawn.go:204, proc_darwin.go:20-25 / proc_linux.go:23-28). But the CRASH-RECOVERY path does not: `RestoreOrCleanupOnBoot` (restore.go:29-45) reads the recorded `e.RunningPID` and calls `killIfAlive`, which does `os.FindProcess(pid)` then `p.Kill()` (restore.go:69) — a POSITIVE-pid SIGKILL that kills only that one process, not the group. The recorded PID is `inner.PID()` (app/sandbox/spawn.go:74 SetEnvRunningPID(envID, inner.PID())), which is the DIRECT child the backend spawned — for python MCP servers that's `uvx` (python.go:75-93 launches `uvx <pkg>`), for node MCP servers that's `npx`. uvx/npx then fork the real `python`/`node` server as a grandchild in the SAME process group. So when the backend crashed leaving a survivor and the next boot runs RestoreOrCleanupOnBoot, it kills the uvx/npx wrapper but the python/node grandchild (the actual MCP server, the memory holder) survives as an orphan reparented to init — and since the manifest row is then cleared (restore.go:40), it is never revisited.
- **修法**: In killIfAlive (restore.go), kill the process GROUP, not the bare pid: on unix do `syscall.Kill(-pid, SIGKILL)` (the recorded child is the group leader since spawn sets Setpgid, so pgid == pid), with a positive-pid `p.Kill()` fallback if the group is already gone; on windows use the existing taskkill /T path.
